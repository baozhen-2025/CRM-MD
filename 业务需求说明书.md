# 主数据集成管理平台 业务需求说明书

**文档版本**: V1.0
**编写日期**: 2026-03-19
**项目名称**: 一级公司与三级公司主数据双向同步系统

---

## 1. 业务背景

### 1.1 现状分析

集团采用分级管理模式，存在一级公司（集团总部）和三级公司（子公司）两个管理层级。各级公司均有独立的主数据管理系统，存在以下问题：

| 问题 | 影响 |
|------|------|
| 数据孤岛 | 各级公司数据独立维护，缺乏统一标准 |
| 数据不一致 | 同一客商在不同系统中编码、名称不一致 |
| 重复维护 | 相同数据在多个系统中重复录入和维护 |
| 数据口径差异 | 统计报表数据来源不一致，影响决策分析 |

### 1.2 业务目标

建立一级公司与三级公司之间的主数据双向同步机制，实现：

1. **数据统一**：确保两级公司主数据的一致性
2. **减少重复**：避免相同数据的重复录入和维护
3. **追溯透明**：数据变更可追溯，来源可查
4. **协同高效**：提升跨级数据协同效率

## 3. 业务场景

### 3.1 场景一：一级公司下发给三级公司

#### 3.1.1 场景描述

一级公司（集团总部）作为数据权威来源，将标准化的主数据下发给三级公司。

#### 3.1.2 适用数据

| 数据维度 | 下发方式 | 说明 |
|----------|----------|------|
| 组织架构 | 全量下发 | 集团统一管理组织架构，下发到各子公司 |
| 客商 | 增量下发 | 集团新增的客商信息，同步给各子公司 |
| 人员 | 按需下发 | 集团总部人员信息，按需同步 |

#### 3.1.3 业务流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                    场景1：下发同步业务流程                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  一级公司                              三级公司                      │
│  ────────                              ────────                      │
│                                                                      │
│  1. 维护主数据                                                       │
│     (新增/修改)                                                      │
│         │                                                            │
│         ▼                                                            │
│  2. 数据推送到中间平台                                               │
│         │                                                            │

│         │  ║                   T+1 同步                          ║    │
│         │  ═════════════════════════════════════════════════════    │
│         │                                                            │
│         └───────────────────────────────────────→ 3. 接收数据       │
│                                                      │               │
```

#### 3.1.4 业务规则

| 规则 | 说明 |
|------|------|
| 数据权威性 | 一级公司下发的数据为权威数据，三级公司以一级公司数据为准 |
| 字段覆盖 | 下发数据覆盖三级公司对应字段（可配置不覆盖字段） |
| 新增自动入库 | 一级公司新增的数据，三级公司自动新增 |
| 变更需审核 | 一级公司变更的数据，如三级公司有冲突，需人工审核 |

### 3.2 场景二：三级公司同步给一级公司

#### 3.2.1 场景描述

#### 3.2.2 适用数据

| 数据维度 | 上报方式 | 说明 |

| 客商 | 直接上报 | 三级公司新增的客商，上报到集团客商库 |
| 人员 | 直接上报 | 三级公司人员信息，上报到集团人员库 |

| 规则 | 说明 |
|------|------|
| 数据完整性 | 上报数据需满足一级公司数据质量要求 |

| 重复校验 | 与集团已有数据重复时，需关联而非新增 |
| 审核机制 | 部分敏感数据需一级公司审核后才能入库 |

## 4. 匹配规则

### 4.1 匹配策略

在数据同步过程中，需要判断源数据与目标数据是否为同一条记录，采用以下匹配策略：

```
┌─────────────────────────────────────────────────────────────────────┐
│                         数据匹配流程                                 │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  源数据 ──→ 按匹配规则查找 ──→ 匹配结果判断 ──→ 执行操作            │
│                                                                      │
│  匹配结果：                                                          │
│  ┌────────────────────────────────────────────────────────────┐     │
│  │ 匹配结果        │ 说明                  │ 处理方式          │     │
│  ├────────────────────────────────────────────────────────────┤     │
│  │ 精确匹配1条    │ 找到唯一对应记录       │ 更新该记录        │     │
│  │ 匹配0条        │ 未找到对应记录         │ 新增记录          │     │
│  │ 匹配多条       │ 找到多个可能记录       │ 人工审核处理      │     │
│  └────────────────────────────────────────────────────────────┘     │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 4.2 匹配条件配置

支持按数据维度配置不同的匹配条件：

#### 4.2.1 组织架构匹配条件

| 优先级 | 匹配规则 | 匹配条件 | 说明 |
|--------|----------|----------|------|
| 1 | 编码匹配 | 组织编码 = 组织编码 | 最精确 |
| 2 | 名称+上级匹配 | 组织名称 = 组织名称 AND 上级编码 = 上级编码 | 编码不同时使用 |

#### 4.2.2 客商匹配条件

| 优先级 | 匹配规则 | 匹配条件 | 说明 |
|--------|----------|----------|------|
| 1 | 编码匹配 | 客商编码 = 客商编码 | 最精确 |
| 2 | 信用代码匹配 | 统一社会信用代码 = 统一社会信用代码 | 编码不同时使用 |
| 3 | 名称匹配 | 客商名称 = 客商名称 | 最后备选 |

#### 4.2.3 人员匹配条件

| 优先级 | 匹配规则 | 匹配条件 | 说明 |
|--------|----------|----------|------|
| 1 | 工号匹配 | 工号 = 工号 | 最精确 |
| 2 | 身份证匹配 | 身份证号 = 身份证号 | 工号不同时使用 |

## 6. 数据优先级与冲突处理

### 6.1 数据优先级

| 场景 | 优先级规则 | 说明 |
|------|------------|------|
| 下发同步 | 一级公司数据优先 | 集团数据为权威数据 |
| 上报同步 | 一级公司数据优先 | 如有冲突，保留一级公司数据 |
| 双向同步 | 按时间戳判断 | 最新修改的数据优先（可配置） |

### 6.2 冲突处理策略

| 冲突类型 | 处理策略 | 说明 |
|----------|----------|------|
| 字段冲突 | 配置覆盖规则 | 指定哪些字段以哪方为准 |
| 编码冲突 | 关联而非新增 | 将两条数据建立关联关系 |
| 删除冲突 | 人工确认 | 一方删除一方修改时，需人工确认 |

## 7. 人工审核

### 7.1 需人工审核的场景

| 场景 | 说明 |
|------|------|
| 匹配到多条 | 自动匹配找到多个可能的对应记录 |
| 无法匹配 | 按规则无法找到对应记录，且不确定是否新增 |
| 数据冲突 | 同一字段双方数据不一致，无法自动判断 |
| 质量问题 | 数据不符合质量要求，需人工确认 |

### 7.2 审核操作类型

| 操作 | 说明 | 适用场景 |
|------|------|----------|
| 新增为新记录 | 将源数据作为新记录入库 | 确认是新数据 |
| 绑定已有记录 | 将源数据与已有记录关联 | 确认是同一条数据 |
| 跳过 | 不处理该条数据 | 暂时不需要处理 |
| 拒绝 | 拒绝该数据入库 | 数据不符合要求 |

## 8. 数据同步频率

### 8.1 同步策略

| 同步类型 | 频率 | 说明 |
|----------|------|------|
| 增量同步 | T+1（每日） | 同步前一天的新增和变更数据 |

| 客商同步 | 每日 03:00 | 业务低峰期 |
| 人员同步 | 每日 04:00 | 业务低峰期 |

## 9. 数据追溯

### 9.1 追溯要求

| 追溯内容 | 保留期限 | 说明 |
|----------|----------|------|
| 原始数据 | 永久 | 保留同步过来的原始数据 |
| 变更历史 | 至少1年 | 记录每次变更的前后值 |
| 审核记录 | 永久 | 保留所有人工审核的操作记录 |
| 同步日志 | 至少3个月 | 保留每次同步的执行日志 |

### 9.2 可追溯信息

| 信息 | 说明 |
|------|------|
| 数据来源 | 数据来自一级公司还是三级公司 |
| 同步批次 | 数据是哪次同步任务处理的 |
| 变更时间 | 数据何时发生了变更 |
| 变更内容 | 变更前后具体的值 |
| 操作人员 | 人工审核时的操作人员 |

## 10. 成功指标

### 10.1 业务指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 数据一致性 | ≥99% | 两级公司主数据一致的比例 |
| 自动匹配率 | ≥95% | 无需人工审核即可匹配成功的比例 |
| 同步及时性 | T+1 | 数据变更后第二天完成同步 |
| 人工审核及时性 | ≤24小时 | 待审核数据在24小时内处理完成 |

### 10.2 技术指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 同步成功率 | ≥99% | 每次同步任务的成功率 |
| 数据准确率 | 100% | 同步数据无丢失、无错误 |
| 系统可用性 | ≥99.9% | 系统正常运行时间占比 |

## 11. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 网络中断 | 同步失败 | 自动重试机制，异常告警 |
| 数据量激增 | 同步超时 | 分批处理，优化性能 |
| 匹配规则不合理 | 大量人工审核 | 持续优化匹配规则 |
| 接口变更 | 同步失败 | 接口版本管理，变更通知机制 |
